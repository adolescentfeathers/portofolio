Getting Started

You can instal Git in several different ways :

  • by using one of the GUI installers
  • by using a package management system
  • by compiling it from source

Configuring Git
Configure Git to know who you are. git config --global user.name "Your Name" git config --global user.email "user@domain.com" Set the Git user for a specific Repository cd /path/to/Repository git config user.name "Your Name" git config user.email "user@domain.com" Turn colors on wherever possible in the Git UI git config --global color.ui auto Configure Git's editor. git config --global core.editor /path/to/editor

Creating a New Repository
Repositories in Git are stored on local file system right alongside the code they track. Create a repository by typing git init in the directory that you want to start tracking files in.

Create a repository
mkdir some-repository cd some-repository git init
Create a repository in an existing directory, and add all files from that directory.
cd /path/to/some/directory git init git add . git commit -m "some commit message"

Creating a Local Copy of an Existing Repository

Create a clone to remote repository to start making changes to it. The git clone command initializes a new repository local computer and fetches the entire history--all changes that have been tracked during the life of that repository.

Clone a repository
git clone some-repository

Clone a repository into a specific path
git clone some-repository some-path

Create a shallow clone with the last fifty commits
git clone --depth 50 some-repository

These are valid Git repository URls.
user@ssh host::path_to_repo git://some_domain/path_to_repo http://some_domain/path_to_repo https://some_domain/path_to_repo/path_to_repo

Working with Git

A handful of commands are needed to get through most tasks.

Seeing What Has Changed

  • What the status of a new repository looks like. git status # on branch main # # Initial commit # nothing to commit (create/copy files and use "git add" to track)
  • What git status looks like in a repository with changes.

    Git status requires a repository with some changes in its working tree to see the various output.

    git status # On branch main # Changes to be committed: # (use "git reset HEAD ..." to unstage) # # modified: castanaut.gemspec # # Changed but not updated: # (use "git add ..." to update what will be committed) # (use "git checkout -- ..." to discard changes in ... # # modified: README.txt # # Untracked files: # (use "git add ..." to include in what will be ... # # pkg/
  • What git status looks like with no changes git status # On branch main nothing to commit (working directory clean)

Staging Changes to Commit

Git uses a two-step process to get changes into the repository. The first step is staging changes through git add. Staging a change adds it to the index, or staging area. This sits between the working tree and the actual repository.

  • Stage an entire file to Commit. git add path/to/file ... or ... git add path/ ... or everything under the current directory ... git add
  • Add all files in the current repository. git add -A|--all
  • Add all tracked files that have been changed. git add -u|--update
  • Choose which changes to commit. git add -p|patch ... or a specific file ... git add -p path/to/file
  • Open the current diff in the editor. git add --e ... or a specific file ... git add -e path/to/file

Committing Changes

Git tracks changes to repository through commits, which make with the git commit command. Each commit requires a commit message. Use -m and a string in quotation marks for message or use Git's editor to write a message. Avoid git add and commit every change in working tree with -a parameter. It commits everyting staged and all the changes to working tree.

  • Stage and commit changes. git add < some file > //remove space git commit -m "some message"
  • Commit all modified files. git commit -m "Some message" -a
  • Commit and launch editor for commit message. git add < some file > //remove space git commit

Ignoring Files

Software projects generate a lot of cruft. Some of it doesn't need to commit.

  • Ignore a specific file and/or path called cache Add the following to .gitignore: cache
  • Ignore all .swp files from Vim. Add the following to .gitignore: *.swp
  • Set up a global excludes file. The excludesfile can exist anywhere on computer. The following example puts it in home directory in the .gitignore file:
    git config --global core.excludesfile \ ~/ .gitignore
